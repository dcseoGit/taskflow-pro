# 00. Overview — 이 문서 세트를 읽는 법

## 프로젝트

**TaskFlow Pro** — 팀 업무 관리 풀스택 웹 앱

---

## 문서 구성 (6개 파일 매핑)

| 파일 | 역할 | 이 파일에서 알 수 있는 것 |
|------|------|--------------------------|
| `00-overview.md` | 문서 안내서 | 전체 문서 구조, 읽는 순서, 분리 원칙 |
| `01-product.md` | 제품 정의 | 왜 만드는가, 누가 쓰는가, 무엇을 하는가 |
| `02-specs.md` | 기술 명세 | 기술 스택, 폴더 구조, API 명세, DB 스키마 |
| `03-design.md` | 설계 결정 | 아키텍처 다이어그램, 계층 역할, 의존성 방향 |
| `04-tasks.md` | 작업 계획 | Phase별 체크리스트, 단계별 검증 방법 |
| `05-conventions.md` | 협업 규칙 | 네이밍, 커밋 형식, 코드 스타일, 금지 사항 |

---

## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (지금)        (목적·범위)    (스택·명세)   (구조·설계)   (할 일)       (규칙)
```

**처음 합류한 사람이라면**: 00 → 01 → 02 순서로 읽으면 "무엇을, 어떻게, 어디서"가 잡힌다.  
**코드를 짜기 전이라면**: 03 → 04 → 05를 반드시 먼저 확인한다.

---

## 왜 6개로 나누는가 — 관심사 분리 원칙

코드에서 하나의 함수가 여러 책임을 지면 읽기 어렵고 수정할 때 부작용이 생긴다.  
문서도 동일하다. **하나의 파일 = 하나의 질문에 대한 답**이라는 원칙을 따른다.

| 질문 | 담당 파일 | 다른 파일이 건드리지 않는 이유 |
|------|-----------|-------------------------------|
| "이게 왜 필요한가?" | `01-product.md` | 기술 결정과 섞이면 목적이 흐려진다 |
| "어떤 기술을 쓰는가?" | `02-specs.md` | 제품 비전과 구현 세부사항은 변경 주기가 다르다 |
| "어떻게 구성되는가?" | `03-design.md` | 설계는 스펙이 확정된 후에야 의미가 생긴다 |
| "지금 무엇을 해야 하는가?" | `04-tasks.md` | 할 일 목록은 매일 바뀌지만 설계는 자주 안 바뀐다 |
| "어떻게 작성하는가?" | `05-conventions.md` | 규칙은 모든 파일에 적용되므로 독립 분리가 맞다 |

> 파일이 많아 보여도 찾아가는 비용이 줄고, 수정할 때 다른 파일을 건드릴 이유가 없어진다.  
> 변경 주기가 다른 것을 한 파일에 합치면 "왜 이게 여기 있지?"라는 혼란이 생긴다.
